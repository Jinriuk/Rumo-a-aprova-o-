## 3. Métricas a coletar e limites (SLO)

Para cada cenário, medir **a coordenação abrindo a Área da Escola** (a soma das
queries Q1–Q5 do `queries-e-indices.md`) e a RPC isolada.

| Métrica | Como medir | Limite (teto) |
|---|---|---|
| Latência `resumo_escola()` (Q1) | `\timing` no psql ou `Date.now()` no teste | 300/500: **< 1s** · 10k: **< 3s** |
| Plano de Q1 sem `Seq Scan` indevido | `explain (analyze, buffers)` | sem scan de tabela multi-tenant inteira |
| Latência `listarAlunos` (Q2) | idem | < 800ms (após RLS, só a escola) |
| Tempo total "abrir Área da Escola" | soma Q1–Q5 (as 5 chamadas do `useRecurso`) | 300/500: **< 2,5s** · 10k: **< 5s** |
| Exportação CSV (alunos) | gerar `linhasRelatorioAlunos` + `paraCSV` | em memória, < 200ms p/ 10k (sem I/O) |
| Paginação da lista (50/pág.) | render só de 50 itens | constante, independe do total |
| Pico de memória do cliente | DevTools → Memory | sem crescimento linear com N (paginação) |

**Critério de aprovação do cenário:** todas as latências dentro do teto **e**
nenhum plano com varredura de tabela multi-tenant inteira **e** isolamento
intacto (a escola de carga nunca aparece para outra escola).

## 4. Roteiro de execução (quando staging existir)

1. Subir staging isolado (projeto Supabase separado do demo).
2. Aplicar migrations + seed padrão (01–12) + `massa_10k.sql`.
3. Rodar EXPLAIN da Q1 (procedimento do `queries-e-indices.md` §3) **antes** de
   qualquer índice novo → registrar plano e tempo (baseline).
4. Medir as latências da §3 (antes).
5. **Se** a Q1 estourar o teto e o EXPLAIN apontar o agregado como gargalo:
   aplicar o índice candidato (`queries-e-indices.md` §4.2) numa branch, medir
   de novo (depois), comparar e medir o **custo de escrita** (INSERT em
   `registros_estudo`).
6. Rodar a suíte de RLS/isolamento contra o staging com massa.
7. Registrar antes/depois no `relatorio-perf1-escala-carga.md`.
8. Derrubar a escola de carga (limpeza §2.2).

## 5. Carga concorrente (fora do escopo de execução agora)

Carga de **concorrência** real (X coordenações + Y alunos lançando registros ao
mesmo tempo, via k6/Artillery contra a API PostgREST/Edge) depende de:
- staging isolado com GoTrue (contas Auth) — hoje só o demo tem Auth;
- chaves de serviço **fora** de qualquer doc/log (regra de segurança).

Fica **planejado e bloqueado** aqui. O que dá para garantir agora — e está
garantido — é a carga de **volume de dados** (massa sintética + EXPLAIN +
latência da query mais pesada), que é o gargalo de escala da escola média.
